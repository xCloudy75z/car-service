Adversarial Design Review

‹ Project hub · six Opus engineers vs. the spec

The exercise: before writing a line of code, six independent Opus engineers were each told to break the design from a different angle — prediction logic, data model, offline/installability, security, accessibility, and future growth. They didn't praise it. They attacked it.

The result: real problems found, several flagged by more than one reviewer independently — which is the strongest signal they're real. Every one has a concrete fix below. Almost all are cheap to fix now and expensive to fix after building.

🔴 Blockers — must fix before building

B1A single odometer typo (52000 → 520000) makes the whole app scream "everything overdue" — forever4 reviewers
What breaks"Current mileage" is the highest odometer you've ever entered. One fat-fingered high number becomes your permanent current mileage, so every prediction turns red. The typo guard only warned about numbers that were too low, so a too-high one sails through — and future correct entries can't undo it.
FixWarn on implausible jumps in both directions; make current-mileage outlier-aware and recomputed from scratch; show the current mileage on screen and let you exclude a bad reading. Also: define "last time X was done" as the highest-odometer entry with that job (not entry order), so back-filled old receipts don't corrupt predictions.
B2Restoring a backup wipes your real history before checking the file is valid2 reviewers
What breaksImport the wrong file (or a truncated/foreign one), tap Confirm, and your genuine history is gone with nothing usable in its place. The confirm dialog guards against an accidental tap — not against a bad file.
FixNever touch your live data until the incoming file is fully checked: read → validate shape → upgrade a copy → validate again → then replace. Reject files that aren't ours or are from a newer app version. Auto-save a snapshot of your current data right before any restore, with an Undo.
B3A malicious/booby-trapped backup file could run code or corrupt the app2 reviewers
What breaksText you type (workshop, notes) rendered the wrong way, or a hand-crafted import file, could inject scripts, poison the app's internals, or crash it on every load.
FixRender all user text as plain text only (never as HTML); strip dangerous keys from imported files; rebuild imported data field-by-field against a strict allow-list; add a strict security policy (CSP) in the page that also mechanically enforces "nothing ever leaves your device."
B4After every update you publish, users stay stuck on the OLD version — sometimes forever2 reviewers
What breaksAn installed offline app aggressively caches itself. Without an explicit update path, a new version installs but never activates — bug fixes never reach the user.
FixStamp each build with a unique version so the browser notices the change; take control immediately and reload once, or show a gentle "Update available" tap; clear out old caches on activation.
B5On GitHub Pages the app installs blank / with missing icons / broken offline1 reviewer
What breaksThe app lives at …github.io/car-service/, not the site root. Any path written from the root (icons, manifest, offline files) points to the wrong place — the classic way this exact recipe fails on GitHub Pages.
FixUse relative paths everywhere (manifest, service worker, icons); register the offline worker relative to the page; add the Apple-specific tags iPhones need for a clean home-screen install. Test under the real sub-path, not just the root.
B6On iPhone, the app can quietly erase itself after ~7 days — and backup is buried2 reviewers
What breaksiOS auto-clears a website's storage after ~7 days of no use unless it's installed to the home screen. You'd realistically open this app around workshop visits — months apart. Come back to an empty app, and the backup button was hidden in Settings you never opened.
FixAsk the browser to keep storage persistent; put backup on the home screen with "Last backed up: N days ago" and a gentle nudge; first-run note explaining the data lives only on this device; clear "Add to Home Screen" guidance on iPhone.
B7The most important signal — due / overdue — is invisible to colour-blind and screen-reader users1 reviewer
What breaksStatus shown by red/amber/green only. ~8% of men can't tell them apart; a screen reader announces nothing. In sunlight it's hard for anyone.
FixEvery status carries an icon and words ("Overdue by 2,000 km", "Due soon", "OK", "Not logged yet") plus a screen-reader label. Bake accessibility into the design from step one, not "final polish."
B8First-run maths shows nonsense (∞ / NaN) and pop-up sheets trap keyboard users2 reviewers
What breaks"Average per year" divides by a near-zero time span on day one → absurd or broken numbers as the first impression. And the slide-in forms don't move keyboard focus, so keyboard/screen-reader users get stuck behind them.
FixFloor the "per year" maths and show "—" until there's enough history; guard every calculation against empty/zero. Give sheets proper focus handling (move focus in, trap it, Esc to close, return focus out).

🟠 Majors — should fix

M1Backfilling old receipts triggers a false "odometer too low" warning every time
FixMake the check date-aware: only warn when mileage disagrees with the entry's date neighbours, so correctly-entered old receipts stay quiet.
M2On a used car, predictions look empty ("not yet recorded") for everything already serviced
Why it mattersYour target user adopts the app for a car that's already years old across mixed workshops. Every job reads "not logged" — even "never done" to a buyer.
Fix (a decision below)Let the user optionally seed "last done at ~X km (before I started logging)" so predictions work from day one.
M3Distance-only predictions call brake fluid "OK" for years on a low-mileage car
Why it mattersBrake fluid/cabin filter are really time-based. A 5,000 km/year driver would be told they're fine when they're overdue by years — on safety items.
Fix (a decision below)At minimum add a soft "also due by time — check your manual" caption so a green light isn't read as all-clear.
M4No Undo on delete; error messages vanish in 3 seconds
FixDelete with a 6-second "Undo" toast instead of relying on a confirm; make storage-failure errors stay until dismissed so you never think something saved when it didn't.
M5"Due soon" fires 10,000 km early for spark plugs (10% of a big interval is huge)
FixCap the "due soon" lead to a sensible absolute distance (~1,000 km) instead of a flat 10%.
M6The data shape makes the top future features (multi-car, photos, sync) a costly rebuild2 reviewers
Why it mattersEntries/intervals sit beside the car instead of inside it (multi-car = migration + rewrites); photos would blow the small storage limit; sync needs per-record IDs and timestamps that don't exist yet.
Fix (a decision below)A few near-free changes now — proper unique IDs + timestamps, intervals as small objects, nesting data under the car — turn future rebuilds into simple additions.

🟡 Minors — polish & robustness

  • Honour "reduce motion" (some people get motion-sick from slide animations).
  • Minimum 44px tap targets on chips and buttons.
  • Colour tokens that pass contrast in both light and dark.
  • Handle "1,250" style number typing (thousands separators).
  • Warn (don't block) on a future-dated service; default the date to today.
  • Stop double-tap creating duplicate entries.
  • Allow an entry with unknown mileage (old receipts), excluded from predictions.
  • Tap a card to read full notes without entering edit mode.
  • Define what search covers (workshop + notes + job) and show a "no results" state.
  • Year/date jump for very long histories.
  • Keep the saved theme in sync after a restore (no colour flash).
  • Robust, step-by-step data upgrades (each tested, never half-applied).
  • Tidy job tags (trim/lowercase) and protect the built-in job names.

🤔 3 decisions I need from you

1. How much future-proofing to bake in now?

A handful of changes make future features (multiple cars, photos, cloud sync) cheap later — but add a little complexity today for things you don't need yet. Recommended: Balanced — do the near-free ones (unique IDs, timestamps, nest data under the car), skip the speculative ones.

2. Add an optional "last serviced before I started logging" anchor?

Fixes the "everything looks never-done" problem on a used car. Recommended: Yes — it's core to being useful on a real, older car from day one.

3. What to do about time-based items (brake fluid, etc.) in v1?

We locked "distance-only" earlier, but it can wrongly say "OK" on safety items for low-mileage drivers. Recommended: keep distance-only + a soft "also due by time" caption; full time rules come next.
