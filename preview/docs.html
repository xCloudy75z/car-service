Car Service History

An offline-first, installable web app that keeps your car's complete service history on your own device and predicts what maintenance is coming up. No login, no server, no framework.

v1 + v1.1 shipped145 testsCI passing Vanilla JSOffline-firstInstallable PWAWCAG-AA

Overview

Car Service History is the single, private source of truth for your car's maintenance: what was done, when, at what mileage, for how much, where — and what's due next. It installs to your phone's home screen, works fully offline, and updates itself in place.

Live app: xcloudy75z.github.io/car-service/app/  ·  Repo: xCloudy75z/car-service

Features

Log & review

  • Add a service: date, odometer (optional), workshop, cost, multiple jobs per visit, notes.
  • Timeline with search + job filters; tap a card for notes; edit/delete with undo.
  • Insights: total & yearly spend, service count, cost-by-job breakdown.

Predict

  • Per-car "next due" with progress bars + soft time reminders.
  • Used-car anchors: "last done at ~X km" so a pre-owned car predicts from day one.
  • Editable intervals: set distances, turn predictions on/off, reset to defaults.

Your garage

  • Multiple vehicles — each with its own history, intervals, custom items, anchors.
  • Switch from the header; add/edit/delete in Settings (with undo; last car protected).
  • Custom maintenance items — your own named jobs with an emoji + optional interval.

Safe & private

  • All data stays in your browser; nothing leaves the device.
  • One-tap Backup to a file; safe Restore (rejects foreign/hostile files; snapshot + undo).
  • Installable + offline; in-app "Check for updates" + a version/build stamp.

How it works

Everything is a single JSON object saved under one localStorage key. When you add a service, the app recomputes predictions and stats on the fly and re-renders — no server round-trip, instant, offline.

Prediction rule: a job is "predicted" for a car precisely when a distance exists for it in that car's intervals. So turning a prediction on or off is just adding or removing a distance, and everything (built-in jobs and your custom ones) works the same way. "Next due" = the mileage of the last time that job was done (or your anchor) plus its interval, compared to your current mileage (the highest odometer you've entered).

Data model (v3)

One key car-service:data holds a versioned, car-nested object:

{
  version: 3,
  activeCarId,
  cars: [{
    id, profile:{name,make,model,year,plate},
    entries:[{ id, date, odometer|null, workshop, cost, tags[], notes,
               createdAt, updatedAt, deletedAt }],
    intervals:{ oil:{km:10000}, brake_fluid:{km:40000, timeHintMonths:24} },
    customJobs:{ "cj_ab12cd34":{ label:"Coolant flush", icon:"❄️" } },
    baselines:{ brake_fluid:{ odometer:35000, date:"2024-05-01" } }
  }],
  settings:{ theme, currencyLabel, lastBackupAt }
}
  • Car-nested — each car owns its data; activeCarId selects one.
  • Predicted = key in intervals; the built-in "predicted" flag is only a seed for new cars.
  • Custom keys match ^cj_[a-z0-9]{4,}$ (backup-safe — not a raw UUID).
  • Migrations run on load (a versioned, idempotent, guarded chain).
  • Soft delete + per-record timestamps — future-proof for sync.

Architecture

Pure, testable logic is separated from DOM and persistence:

schema → format/validate/migrate/calc/select  (PURE, unit-tested, no DOM/clock/random)
       → store  (localStorage + all mutations + IDs + undo + backup)
       → ui/*   (DOM builders, accessible dialog/toast — textContent only)
       → app.js (controller: tabs, sheets, Garage, intervals editor, backup, updates)

A "purity lint" (scripts/lint-pure.js) fails the build if a pure module uses Date.now()/Math.random() or any source file uses innerHTML — so the risky logic stays deterministic and injection-proof. Full detail in docs/ARCHITECTURE.md.

Security & privacy

  • Nothing leaves your device — enforced by a strict Content-Security-Policy with connect-src 'none' (no network calls possible) and zero off-origin resources (fonts are self-hosted).
  • All user text is rendered as plain text (textContent), never as HTML — no injection.
  • Import is treated as hostile: a backup file is parsed with a prototype-pollution-proof reviver, checked for the app marker + version, then rebuilt field-by-field against an allow-list; your live data is snapshotted before anything is replaced, with an undo. A bad file can't run code or wipe your history.
  • A backup file contains your car details/notes — it's the one thing that can leave the device, so keep it private.

Accessibility

Built in from the start: WCAG-AA contrast, status shown as icon and words (never colour alone), a real tab pattern with arrow-key navigation, focus-trapped dialogs that restore focus on close, live-region toasts (errors don't auto-dismiss), role="switch" toggles, ≥44px tap targets, and reduced-motion support.

Build & deploy

No dependencies to install. From the repo root:

node --test "tests/**/*.test.js"   # run the 145 tests
node scripts/lint-pure.js          # purity + safe-DOM lint
node scripts/build.js              # assemble app/ (stamps version + build time)
node scripts/serve.js              # serve app/ at http://localhost:5173

The build copies src/ into app/, hashes every file to a version it stamps into the service worker (cache name) and the Settings "Version" line, and copies the icons. app/ is committed; GitHub Pages serves it at …/car-service/app/. A CI workflow runs tests + lint + build on every push.

Because the version hashes every file, even a CSS-only change bumps it — so the installed app actually updates when you tap "Check for updates".

Testing

145 automated tests (Node's built-in runner) cover predictions/stats, input + hostile-import validation (incl. a lossless export→import roundtrip with custom jobs and anchors), schema migrations, and every store operation. UI is verified in a real browser before each deploy. The purity lint guards determinism and safe DOM.

How it was built

Every feature followed the same disciplined path:

  • Brainstorm → a written spec.
  • Break the spec — independent agents adversarially try to break it; fixes folded in. (This caught a backup data-loss bug and several crashes before any code.)
  • Plan → and break the plan too.
  • Build in small slices, each written test-first, verified running in a real browser, then deployed before the next.
